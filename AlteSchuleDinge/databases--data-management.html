Entity-Relationship (E-R) Diagrams &
Developing PostgreSQL Enterprise Databases

Description
Objectives:
  • Understand concepts of conceptual and logical database design, Entity-Relationship, and Unified Modeling Language (UML) diagrams;
  • Understand the purpose and structure of Data Definition Language (DDL); and
  • Create and populate a PostgreSQL relational database
This project required developing an Entity-Relationship diagram based on the National Park Service fish inventory. The diagram needed to include: a) Entities, b) attributes for each entity and special characteristics associated with each attribute (i.e. primary key or multi-valued), c) relationships and cardinalities, and d) a legend.
Then, it was necessary to create a UML diagram based on the E-R diagram, create Data Definition Language (DDL) statements based on the UML diagram, create a PostgreSQL database using DDL statements, and lastly, populate database. 
Analysis Procedures
The processes for achieving the objectives of this project comprised, first and foremost, a significant attention to detail. This is due to the nature of the project being characterized by numerous points of data, specific attributes, and connecting relationships amongst groups of data amongst tables. It was imperative to organize all of these facets such that the database was free of errors and any users/consumers/connecting entities to the underlying data would not experience issues in functionality.

Developing and Managing File Geodatabases &
Multi-User Enterprise Geodatabases


​Description
This project had two main components to it:
1) to create and apply topological relationships to feature datasets for data management integrity
2) create an enterprise geodatabase (EGDB) in a RDMS – PostgreSQL in this case – and populate it with feature classes originating as SQL queries created by me
​
​The goal of the first part was to apply topology to data and see how the rules work. The goal of the second problem was to see how to create tables in an enterprise geodatabase.
​
Analysis Procedures
​For data management integrity and analyzing spatial relationships, I created a feature dataset topology for the streams polylines, fences polylines, and parkboundary lines feature layers such that rules would be enforced that a) fences must not overlap with stream lines and b) streams must be inside the park boundary. For a), the rule makes sense in the topology because fences are not built in streams. For b), in analysis scenarios needing to determine how many streams and/or the spatial relationships between stream locations and species within the park, having the topology rule in place helps streamline (no pun intended) the workflow by ensuring that geoprocessing tasks are using only data within the park polygon(s).
Snapshot of File Geodatabase organization utilizing a Topology Rules Feature Dataset containing in it the feature classes with rules applied to them.
Results
The results of this assignment were a file geodatabase containing several feature classes inside a feature dataset with topology rules applied to them, a map displaying the topological errors, an enterprise geodatabase populated with both data created from SQL statements and the file geodatabase, and a map exhibit displaying the features inside the enterprise geodatabase.
Reflection
This project was foundational and key to developing competence - and confidence - in navigating the complex, nuanced realm of data management and its applications to developing well organized file GDBs and multi-user EGDBs.
